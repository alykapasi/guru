## 1. Multi-Material Sessions

### 1.1 Schema Changes

The current `sessions` table has a single `material_id` foreign key. Replace it with a junction table.

```sql

    PRIMARY KEY (session_id, material_id)

### 1.2 Session Creation Flow

When a user starts a session, they now select 1..N materials. The session setup endpoint changes:

```python
# POST /sessions/create

Internally:

2. Insert one row per material into `session_materials`
3. Generate checklist using all selected materials as context

### 1.4 Add Material Mid-Session

A new endpoint allows adding a material to an existing session:

```python
# POST /sessions/{session_id}/materials

    material_id: uuid.UUID
```

The frontend shows a `+ Add material` button in the left panel of the study interface. It opens a mini-picker showing the user's library (ready materials only). On confirm, the material is added and the retrieval pool expands immediately for subsequent messages.

### 1.5 Frontend Changes

**Session setup screen:** Replace the single material (passed via URL) with a material picker. User sees their library as a checklist, selects 1+, then sets goal.

**Left panel:** Material chips show all active materials. Each chip has an × to remove (only if >1 material active) and a + button to add more.

**Chat citations:** Each assistant message shows source chips below it — `[Material A]`, `[Material B]` — indicating which materials were retrieved for that response.

## 2. Large File Ingestion

### 2.1 Problem

Current ingestion is sequential and synchronous within a background task:
- Parse entire document
- Generate context prefix for each chunk (1 LLM call per chunk, sequential)
- Embed all chunks
- Write to DB

A 3.6MB PDF might produce 80+ chunks → 80+ sequential LLM calls → 30-60 minutes on a local model.

### 2.2 Solution: Sub-Document Splitting

Split large documents into sub-documents at ingest time, process each independently, and surface them to the user as they complete. This gives the user partial results fast rather than nothing for a long time.

```
Large PDF (80 chunks estimated)
│
├── Sub-doc 1: pages 1–20   → ingested, status: ready in ~5 min
├── Sub-doc 2: pages 21–40  → ingested, status: ready in ~5 min
├── Sub-doc 3: pages 41–60  → ingesting...
└── Sub-doc 4: pages 61–80  → pending...
```

The user can start studying sub-doc 1 while sub-docs 2-4 are still processing.

### 2.3 Schema Changes

```sql
-- Migration: 004_sub_documents.sql

ALTER TABLE materials ADD COLUMN sub_doc_count INT DEFAULT 1;
ALTER TABLE materials ADD COLUMN sub_doc_index INT DEFAULT 0;  -- 0 = not split
ALTER TABLE materials ADD COLUMN parent_material_id UUID REFERENCES materials(id);
ALTER TABLE materials ADD COLUMN page_range INT[];  -- [start_page, end_page]
```

Materials with `parent_material_id IS NOT NULL` are sub-documents. The parent material's status is:
- `splitting` → being split into sub-docs
- `partial` → some sub-docs ready, some still ingesting
- `ready` → all sub-docs ready
- `error` → at least one sub-doc errored

### 2.4 Splitting Logic

```python
SUB_DOC_PAGE_LIMIT = 25   # pages per sub-document
SUB_DOC_CHUNK_LIMIT = 30  # or split by chunk count if page count unavailable

async def maybe_split_and_ingest(material_id, minio_key, pool):
    # 1. Parse with Docling to get page count
    # 2. If pages <= SUB_DOC_PAGE_LIMIT: ingest normally (existing flow)
    # 3. If pages > SUB_DOC_PAGE_LIMIT:
    #    a. Update parent status to 'splitting'
    #    b. Split into N sub-docs
    #    c. For each sub-doc: create a materials row (with parent_material_id)
    #                         upload sub-doc slice to MinIO
    #                         add ingest task to queue
    #    d. Update parent status to 'partial'
    #    e. As each sub-doc finishes: check if all done → update parent to 'ready'
```

### 2.5 Retrieval with Sub-Documents

Retrieval works transparently. When a user adds a material to a session, we resolve it to all its sub-documents:

```python
async def resolve_material_ids(material_ids: list[uuid.UUID], pool) -> list[uuid.UUID]:
    """Expand parent material IDs to include all sub-document IDs."""
    async with pool.acquire() as conn:
        rows = await conn.fetch(
            """
            SELECT id FROM materials
            WHERE id = ANY($1::uuid[])
               OR parent_material_id = ANY($1::uuid[])
            AND status = 'ready'
            """,
            material_ids
        )
    return [r['id'] for r in rows]
```

### 2.6 Task Queue: Switch from BackgroundTasks to ARQ

`FastAPI.BackgroundTasks` has no retry, no persistence across restarts, and no concurrency control. For V0.2 swap to **ARQ** (async Redis queue, lightweight, Python-native).

```toml

"arq>=0.26.0",
"redis>=5.0.0",
```

Add Redis to docker-compose:

```yaml

  image: redis:7-alpine
  ports:
    - "6379:6379"
```

Worker definition:

```python

from arq import cron
from app.services.ingestion import maybe_split_and_ingest

async def ingest_task(ctx, material_id: str, minio_key: str):
    pool = ctx['db_pool']
    await maybe_split_and_ingest(uuid.UUID(material_id), minio_key, pool)

class WorkerSettings:
    functions = [ingest_task]
    on_startup = startup  # initialise DB pool
    on_shutdown = shutdown
    max_jobs = 4          # process 4 ingestions concurrently
    job_timeout = 3600    # 1 hour max per job
```

```bash

For handwritten notes specifically, add a second vision pass:

## 4. Mastery Tracking — Hybrid IRT + LLM Signals

### 4.1 Architecture Overview

The mastery system has three layers, each producing a signal that feeds into a final `theta` (ability) score per concept:

```
┌─────────────────────────────────────────────────────┐
│                  Mastery Score (θ)                  │
│              0.0 ──────────────── 1.0               │
└──────────────┬──────────────┬──────────────┬────────┘
               │              │              │
          60% weight     30% weight     10% weight
               │              │              │
    ┌──────────┴─────┐ ┌──────┴──────┐ ┌────┴────────┐
    │   IRT Signal   │ │ LLM Signal  │ │ Self-Report │
    │  Quiz-based    │ │ Chat-based  │ │   (0-4)     │
    └────────────────┘ └─────────────┘ └─────────────┘
```

### 4.2 Schema Changes

Replace the simple `mastery_scores` table with a richer one, and add interaction logging for future DKT training:

```sql
-- Migration: 005_mastery_v2.sql

-- Drop old table
DROP TABLE mastery_scores;

-- New mastery table with IRT parameters
CREATE TABLE mastery_scores (
    user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    material_id     UUID NOT NULL REFERENCES materials(id) ON DELETE CASCADE,
    concept         TEXT NOT NULL,

    -- IRT ability estimate (continuous, ~-3 to +3 internally, mapped to 0-1 for display)
    theta           FLOAT NOT NULL DEFAULT 0.0,

    -- Component signals (stored separately for transparency + DKT training data)
    irt_score       FLOAT NOT NULL DEFAULT 0.5,   -- 0 to 1
    llm_score       FLOAT NOT NULL DEFAULT 0.5,   -- 0 to 1, null until first chat assessment
    self_score      FLOAT,                         -- 0 to 1, null until first self-report

    -- Metadata
    attempts        INT NOT NULL DEFAULT 0,
    last_quiz_at    TIMESTAMPTZ,
    last_chat_at    TIMESTAMPTZ,
    last_updated    TIMESTAMPTZ DEFAULT NOW(),

    PRIMARY KEY (user_id, material_id, concept)
);

-- Interaction log — every quiz question answered, stored for future DKT training
-- This is the dataset DKT will train on when you have enough users
CREATE TABLE mastery_interactions (
    id              UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    material_id     UUID NOT NULL REFERENCES materials(id) ON DELETE CASCADE,
    concept         TEXT NOT NULL,
    session_id      UUID REFERENCES sessions(id),

    -- Interaction type
    interaction_type TEXT NOT NULL,  -- 'quiz_mcq', 'quiz_short', 'chat_assessment', 'self_report'

    -- For quiz interactions
    question_text   TEXT,
    difficulty      FLOAT,           -- 0.0 to 1.0 (easy=0.33, medium=0.66, hard=1.0)
    correct         BOOLEAN,
    score           FLOAT,           -- 0.0 to 1.0

    -- IRT state at time of interaction (snapshot for DKT training)
    theta_before    FLOAT,
    theta_after     FLOAT,

    created_at      TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX mastery_interactions_user_concept_idx
    ON mastery_interactions(user_id, concept, created_at);
```

> **DKT note:** `mastery_interactions` is the training dataset for DKT. Every interaction is logged with the θ before and after. When you have N users with M interactions each, you train DKT on sequences of (concept, difficulty, correct) → predict next θ. The schema is designed for this from day one.

### 4.3 IRT Implementation (1-Parameter Logistic Model)

The 1PL IRT model (Rasch model) is the right starting point — simple, interpretable, works with small data.

```python
# app/services/mastery.py

import math

# Difficulty β values per question difficulty label
DIFFICULTY_BETA = {
    'easy':   -1.0,
    'medium':  0.0,
    'hard':    1.0,
}

# Learning rate — how much a single observation shifts θ
LEARNING_RATE = 0.3

def irt_probability(theta: float, beta: float) -> float:
    """P(correct | theta, beta) using 1PL logistic model."""
    return 1.0 / (1.0 + math.exp(-(theta - beta)))

def update_theta(theta: float, beta: float, correct: bool) -> float:
    """
    Update ability estimate θ based on one observation.
    Uses simplified gradient step on log-likelihood.
    correct=True  → θ increases (more if question was hard)
    correct=False → θ decreases (more if question was easy)
    """
    p = irt_probability(theta, beta)
    observed = 1.0 if correct else 0.0
    # Gradient of log-likelihood: (observed - p)
    delta = LEARNING_RATE * (observed - p)
    return theta + delta

def theta_to_score(theta: float) -> float:
    """Map θ (roughly -3 to +3) to display score (0 to 1)."""
    return 1.0 / (1.0 + math.exp(-theta))

def score_to_theta(score: float) -> float:
    """Inverse: map display score back to θ."""
    score = max(0.001, min(0.999, score))
    return math.log(score / (1 - score))
```

### 4.4 LLM Signal — Chat Assessment

Every N chat turns (default: every 5 assistant messages, configurable), run a background assessment of the conversation to extract concept-level confidence signals.

```python
# app/services/mastery.py

CHAT_ASSESSMENT_INTERVAL = 5   # assess every N assistant messages

async def assess_chat_mastery(
    session_id: uuid.UUID,
    material_id: uuid.UUID,
    concepts: list[str],
    pool,
) -> dict[str, float]:
    """
    Read recent chat messages and assess understanding per concept.
    Returns {concept: score} where score is 0.0 to 1.0.
    Only concepts mentioned in the conversation are scored.
    """
    async with pool.acquire() as conn:
        messages = await conn.fetch(
            """SELECT role, content FROM messages
               WHERE session_id=$1
               ORDER BY created_at DESC LIMIT 20""",
            session_id
        )

    if not messages:
        return {}

    conversation = "\n".join(
        f"{m['role'].upper()}: {m['content'][:300]}"
        for m in reversed(messages)
    )
    concepts_str = ", ".join(concepts[:20])

    prompt = (
        f"Analyse this learning conversation and assess the student's understanding "
        f"of each concept they discussed.\n\n"
        f"Concepts to assess: {concepts_str}\n\n"
        f"Conversation:\n{conversation}\n\n"
        f"For each concept that appeared in the conversation, rate the student's "
        f"demonstrated understanding from 0.0 (confused/wrong) to 1.0 (clear mastery).\n"
        f"Only include concepts that were actually discussed.\n"
        f"Respond with JSON only: {{\"concept_name\": score, ...}}"
    )

    from app.services.llm import _chat, FAST
    import json, re
    raw = await _chat(FAST, prompt, max_tokens=300)
    raw = raw.strip().removeprefix("```json").removeprefix("```").removesuffix("```").strip()
    try:
        return json.loads(raw)
    except Exception:
        match = re.search(r'\{.*?\}', raw, re.DOTALL)
        return json.loads(match.group()) if match else {}
```

### 4.6 Score Aggregation

The final displayed score is a weighted combination:

```python
WEIGHTS = {'irt': 0.6, 'llm': 0.3, 'self': 0.1}

def aggregate_score(
    irt_score: float,
    llm_score: float | None,
    self_score: float | None,
) -> float:
    """
    Weighted aggregation. If a signal is missing, redistribute its
    weight proportionally to the signals that exist.
    """
    signals = {'irt': irt_score}
    if llm_score  is not None: signals['llm']  = llm_score
    if self_score is not None: signals['self'] = self_score

    total_weight = sum(WEIGHTS[k] for k in signals)
    return sum(
        (WEIGHTS[k] / total_weight) * v
        for k, v in signals.items()
    )
```

### 4.7 Mastery Update Flow (full)

```
Quiz submitted
    │
    ├── For each question:
    │       get θ_before for concept
    │       get β from question difficulty
    │       update θ with IRT (update_theta)
    │       log to mastery_interactions
    │       update irt_score = theta_to_score(θ)
    │
    ├── Aggregate: final_score = aggregate_score(irt, llm, self)
    │
    └── Write to mastery_scores

Chat message sent (every 5th assistant message)
    │
    └── Background: assess_chat_mastery()
            → update llm_score for touched concepts
            → re-aggregate final_score

Lesson generated → prompt self-report
    │
    └── POST /profile/mastery/self-report
            → update self_score
            → re-aggregate final_score
```

## 6. Key Patterns to Follow

**Never hold a DB connection while calling the LLM.** Acquire → read → release → LLM call → acquire → write → release. This was the source of most V0.1 bugs.

**Always `str(user["id"])` when passing to asyncpg queries.** asyncpg UUID codec is inconsistent — string is always safe.

**Strip markdown fences from all LLM JSON responses.** Every JSON parse should be wrapped in the standard strip + regex fallback pattern established in V0.1.

**The `mastery_interactions` table is sacred.** Every quiz answer, chat assessment, and self-report must be logged here, even if the score update fails. This is your future DKT training data. Write to it first, update scores second.

## 7. DKT Migration Path

When you have sufficient data (rough target: 50+ users, 500+ interactions per concept), the DKT upgrade works as follows:

1. Export `mastery_interactions` as training sequences: `[(concept, difficulty, correct, theta_before)]` per user
2. Train a 1-layer LSTM (PyTorch, ~100 lines) on these sequences
3. Replace `update_theta()` with `dkt_predict_theta(interaction_sequence)` 
4. Keep IRT as a fallback for new users with <20 interactions (cold start)
5. The `mastery_interactions` log and the `theta` column are unchanged — DKT just produces better `theta` values

The schema is designed so this replacement is localised to `mastery.py`. Nothing else changes.
